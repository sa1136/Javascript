  1. üí° Call Stack, Memory, and Code Execution: JavaScript uses a single-threaded call stack to execute code. The memory heap stores variables and objects, while the call stack processes functions in a Last-In-First-Out (LIFO) manner. When a function is called, it‚Äôs pushed onto the stack and removed once completed.
  2. üåé Global Execution Context: When JavaScript code is run, the engine creates a Global Execution Context. This context has two phases: the creation phase (variables and functions are set up in memory) and the execution phase (code is executed line by line).
  3. ü™Ñ Variable Hoisting: In JavaScript, variable declarations are moved (or "hoisted") to the top of their scope during the creation phase. However, variables declared using var are initialized with undefined, while let and const are not initialized until execution.

    Sample Code for Hoisting: 
    // Example of var hoisting
  console.log(hoistedVar); // undefined
  var hoistedVar = "I am hoisted!";
  
  // Example of let hoisting
  // console.log(hoistedLet); // ReferenceError: Cannot access 'hoistedLet' before initialization
  let hoistedLet = "I am not hoisted in the same way!";
  4. üîÅ Function Hoisting: Function declarations are hoisted along with their definitions, meaning they can be called before they are defined in the code. However, function expressions assigned to variables are hoisted without their assigned values.
  5. üèóÔ∏è Hoisting with let and const: Variables declared with let and const are hoisted but placed in a temporal dead zone. They cannot be accessed before the declaration line, ensuring safer variable usage compared to var.

    • Array Functions: Array methods are preferred for their readability, conciseness, and ease of use, making code simpler and easier to maintain.
  8. üîë Objects: An object is a collection of key-value pairs. Each key (or property) is a string, and the value can be any data type, including other objects or functions. Objects are used to represent structured data and can contain properties, methods (functions), and even nested objects.
  9. ü™û this keyword: The this in JavaScript is not a built-in method, but rather a special keyword. It is used to refer to the context in which the function or method is being executed. The value of this changes based on how the function is called.
  10. üîÇ For...in and For...of Loops: for...of is used primarily for arrays, and for...in is used primarily for objects.
  11. üèõÔ∏è Classes: A class is a blueprint, and an object is what you create from that blueprint.
  12. üéâ Constructor: A special method for assigning properties, automatically called when creating an object.
  13. üìú Inheritance: Allows one class (child class) to inherit properties and methods from another class (parent class). This helps create a hierarchy of classes where the child class can access the functionality of the parent class and add or override its own functionality.
  14. üß¨ Prototypes: Prototypes are a mechanism that allows objects to inherit properties and methods from other objects. Every JavaScript object has an internal property called [[Prototype]], which links to another object called its prototype.
  15. üîí Closure: A closure in JavaScript is a function that "remembers" the variables from its outer scope even after the outer function has finished executing.
      Why Use Closures?
    • Encapsulation: Keep variables private.
    • Data Persistence: Maintain state across function calls.
    • Function Factories: Create dynamic functions with different behaviors.
  16. üöö Currying: Currying is a functional programming technique in JavaScript where a function with multiple arguments is transformed into a sequence of functions, each taking a single argument.
  17. üìû Call, Apply, and Bind:
    • Call: Takes a function from one object and allows it to be used with another object.
    • Apply: Works just like call(), but arguments are passed as an array.
    • Bind: Binds an object to a function and returns a new copy of that function. The new function remembers the this value from the object it was bound to.
  18. üìç Lexical Environment: Local memory + lexical environment of the parent. The lexical environment refers to the context in which a piece of code is evaluated and executed, specifically determining how variables and functions are stored and accessed.
  19. üì¶ Block: Use of multiple statements in a single space where JavaScript expects only one statement.
    • Block Scoped: All functions and variables declared with let and const are block-scoped.
  20. üë• Shadowing: Shadowing happens when a variable in a local scope (e.g., inside a function or block) has the same name as a variable in an outer scope. The inner variable "shadows" the outer variable, taking precedence within its scope.
    • let does not respect block scope.
    • You cannot shadow a let with var (illegal shadowing).
    • You can shadow a var with let.
  21. üìû Callback Function: A callback function in JavaScript is a function that is passed as an argument to another function and is executed later, usually after an asynchronous operation completes or an event occurs.
  22. üîÑ Event Loop: The event loop manages asynchronous operations in JavaScript. It ensures non-blocking execution by checking the call stack and executing tasks from different queues.
      Execution order:
    • Runs all synchronous code in the call stack.
    • Executes all microtasks (Promises, MutationObservers).
    • Processes macrotasks (setTimeout, I/O operations, etc.).
    • Repeats the cycle.
  23. üéØ Callback Queue (Macrotask Queue): Stores low-priority asynchronous tasks, including:
    • setTimeout, setInterval
    • setImmediate (Node.js)
    • I/O operations (e.g., file reading, network requests)
    • UI rendering tasks
    Macrotasks run only after all microtasks are completed.
  24. ‚ö° Microtask Queue:
    • Promises: When resolved/rejected, .then() or .catch() moves to the microtask queue.
    • MutationObserver: Detects DOM changes and executes its callback as a microtask.
    • Since microtasks run first, Promises and MutationObservers execute before any setTimeout or I/O operations.
  25. üéâ Promises & MutationObserver in the Microtask Queue:

  26. üöÄ Memoization: Memoization is an optimization technique that stores the results of expensive function calls and returns the cached result when the same inputs occur again.
      How It Works:
    • A function is wrapped inside another function that maintains a cache (usually an object or a Map).
    • Before executing the function, it checks if the result is already in the cache.
    • If found, it returns the cached result instead of re-executing the function.
    • If not found, the function runs, stores the result in the cache, and then returns the output.
